## 一、系统架构

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **整体架构** | 项目是什么 | 前端（React+TypeScript+Vite/TanStack Start）+ 后端（Python FastAPI + 阿里百炼Qwen）；用户扮演妈妈与AI扮演的抑郁青少年「陈晓」对话，系统实时评分 | 前后端完全分离，前端只调用后端HTTP接口，API Key不暴露在前端 |
| **整体架构** | 架构演进路径 | V1.0单体Prompt → V2.0双Agent解耦（演员+裁判分离）→ V2.0.1格式Bug修复 → 放宽评分+边界条款+推荐话术改为AI自生成 | 旧版单体的核心死穴：角色记忆污染、场景健忘、注意力稀释。新架构核心原则：裁判管「发生了什么」，Python管数学，演员管「怎么演出来」 |
| **双Agent核心流** | 每轮用户输入如何处理 | `用户输入 → judge_agent(两步) → update_ledger(Python唯一计分) → actor(演员生成台词) → 返回前端` | 裁判temperature=0.1（第一步）、0.6（第二步）；演员用qwen-plus；复盘报告用qwen-max |
| **裁判第一步** | JUDGE_PROMPT_1职责 | 识别维度（靠近/倾听/共情/引导）、quality（great/good/neutral/bad/terrible）、coherence（normal/pivot/irrelevant/unsafe）、delta0（初步得分建议） | scoring.md在启动时一次性加载进JUDGE_PROMPT_1和JUDGE_PROMPT_3；改了scoring.md要重启才生效 |
| **裁判第二步** | JUDGE_PROMPT_2职责 | 综合历史、账本备注、第一步结果，输出reason（一句话解释）和attribution_tags（孩子主观感受标签数组） | attribution_tags是JSON字符串数组，不是字符串；喂给演员前必须用`_format_attribution_tags()`清洗成`[标签1][标签2]`格式，否则Python list repr会泄露进prompt |
| **Python账本** | 唯一计分入口 | `update_ledger()`：good/great → 封顶于该维度max；bad/terrible → 不设上限直接生效；neutral → 0分；coherence=pivot → 强制-6；irrelevant → -3；unsafe → 触发熔断 | 好封顶、差不封顶是有意设计（信任更难赚、更容易丢）。GROUP_LIMITS/get_group_earned已废弃，现在每个维度直接用自己的max |
| **演员prompt** | 传入字段及处理顺序 | 演员收到：`user_message_str`（先处理）→ `attribution_tags`（感受定性）→ `new_score`（档位/力度）→ `last_chen`（连续性校验+防重复）→ `actor_scene`（背景细节）→ 生成台词 | 处理顺序从核心内容→外围细节；「别重复措辞」和「场景别雷同」这类否定式约束要放在靠近生成位置（prompt末尾），利用近因效应 |
| **前端** | 技术栈 | React 19 + TypeScript + Vite + TanStack Router + Tailwind CSS；框架是TanStack Start模板但SSR功能基本未使用，实际是静态SPA部署 | `createServerFn`只存在于example.functions.ts示例文件，未被任何业务代码引用。前端build命令是`npm run build`，产物在dist/ |

## 二、接口设计

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **后端接口** | `/chat` 接口 | `POST /chat`，Body：`{ message, history, current_score, ledger }`，返回：`{ history, new_score, ledger, status }` | ledger必须在每轮请求中传入并将返回的updated_ledger存回前端State，否则账本状态无法持久化（这是之前的重大Bug之一） |
| **后端接口** | `/report` 接口 | `POST /report`，生成复盘报告，传入完整对话历史+账本，调用qwen-max | 报告prompt硬编码在report_api()函数里，不从文件读取 |
| **后端接口** | 健康检查 | `GET /`，返回`{"message": "hello world"}`，用于判断后端是否正常启动 | 故障排查第一步：先访问/，如果hello world正常而业务接口异常，问题在业务代码；如果hello world也异常，问题在容器/服务启动阶段 |
| **前端调用** | API地址配置 | 使用环境变量 `import.meta.env.VITE_API_BASE_URL`，本地开发在`.env`文件中设置 | 每次改`.env`后必须重启Vite dev server才生效；前端打包时VITE_*变量会被烧入dist产物，所以换环境后必须重新build |
| **CORS** | 跨域配置 | FastAPI中用`CORSMiddleware`，测试阶段`allow_origins=["*"]`；生产环境应指定具体域名 | 前后端不同域名时必须配CORS，否则浏览器会报blocked by CORS policy |

## 三、部署（线上）

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **服务器选型** | 中国大陆可访问 | 香港轻量服务器（腾讯云/阿里云）：无需备案、大陆可访问，延迟略高但对AI聊天影响小 | 国内服务器直接访问OpenAI/阿里百炼：香港服务器没问题；如果要用大陆服务器调用OpenAI需要中转API（如OpenRouter、硅基流动等） |
| **Nginx配置** | 静态前端+FastAPI反代 | 前端dist托管在`/var/www/项目名/`；Nginx location `/` → 静态文件，location `/demoapi/` → proxy_pass → FastAPI端口 | 前端如果请求路径不对（如直接请求`/chat`而不是`/demoapi/chat`），Nginx找不到静态文件会返回404，此时Server头是nginx而不是uvicorn——这是路径配置问题而非后端问题 |
| **公司服务器结构** | wwwroot+Docker架构 | 前端文件放`wwwroot/demo-xxx/`下（保留userinit等系统文件，只覆盖index.html和assets/）；后端Python文件放`user/src/app/`下；`run.sh`执行`python App.py`启动后端；重启Docker容器使改动生效 | 更新已有Demo：覆盖前端+覆盖后端文件+重启Docker。新建Demo：需复制整个前后端目录，新建Docker容器，申请新域名。两个Demo之间完全隔离，改一个不影响另一个 |
| **后端持久化** | nohup vs systemd | 测试阶段用`nohup uvicorn App:api --host 0.0.0.0 --port 8000 > backend.log 2>&1 &`；生产环境建议改为systemd服务 | nohup进程在服务器重启后会消失；日志文件路径在`~/项目名/uvicorn.log`或backend.log |
| **部署顺序** | 先后端后前端 | ①部署后端并验证`/`接口返回hello world → ②将后端域名写入前端`.env`（VITE_API_BASE_URL） → ③`npm run build` → ④上传dist到服务器 | 前端build后API地址已烧入产物，换后端地址必须重新build；不能先部署前端再改API地址 |
| **内网穿透（开发测试）** | 给公司10-20人临时测试 | cpolar：下载对应CPU架构版（M系Mac用arm64，Intel用x86_64）；需要先`xattr -d com.apple.quarantine`绕过GateKeeper；`./cpolar http 8000`暴露后端，`./cpolar http 8080`暴露前端 | 在NYU网络使用会触发学校安全监测（已发生）；cpolar地址每次重启会变，所以需要同时更新前端.env并重新build才能生效 |

## 四、角色与评分设计

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **陈晓人设** | 角色核心设定 | 16岁上海女生，轻度抑郁，因数学老师当众羞辱（「你连这种题都不会，简直太笨了」）产生进教室的生理应激；跟妈妈说过「我最近很难受」，妈妈回「睡一觉就好了」后产生习得性无助；喜欢画画——握笔时唯一感到有价值 | 陈晓的反应必须从这几条具体史实出发，比喻（飞鸟等意象）如果与史实无关会让对话显得通用化、不像「这个具体的陈晓」 |
| **陈晓人设** | 对话开口机制 | 矛盾的求助意愿：希望被问「你怎么了」然后等她慢慢说完；如果问完接的是建议或否定，比没被问还难受。被正确回应2-3次才会真正松动；「但是」之后前面所有验证归零 | 情绪变化不是「决堤」而是「裂缝」——即使高分阶段也只开口1-2句，语气依然平淡，不哭花妆也不滔滔不绝 |
| **四维度** | 评分维度体系 | **靠近**（不强迫陪伴，发现信号+低压力叩门）→ **倾听**（捕捉情绪，情绪命名+开放式提问+反映核心感受）→ **共情**（站在孩子侧，不接「但是」+承认盲区+给予肯定）→ **引导**（还权给孩子，委婉建议+给出选择权） | 四维度来自领导重写，对应旧的五阶段十维度；旧十维度名称依然是`init_ledger`里的key，四维度体现在scoring.md的分组标题和JUDGE_PROMPT的分类逻辑中 |
| **评分范围** | 得分区间设计 | 初始值35分；各维度有独立max（靠近/倾听/共情/引导分别约17/21/18/9=65分可得）；加分档封顶于维度max；扣分不封顶；0熔断，100通关 | 「好封顶、差不封顶」是有意设计（信任难赚易失）；评分放宽后总可得分约80分，给对话留了更多容错空间 |
| **学术支撑** | 各维度理论依据 | 靠近←心理阻抗理论；倾听←Lieberman情感标注+Carl Rogers 1957共情理解；共情←DBT隐性无效化+乔哈里视窗；引导←CBT行为激活技术 | Rogers 1957核心是「共情性理解是治疗起效的必要条件」；「产生被看见的安全感」是后续学派对该理念的阐释，非原文 |
| **推荐话术** | 触顶后推荐下一步 | `generate_recommendation()`算出剩余维度→不再喂给模型具体示例表格，只给维度名+该维度一句话核心原则+最近2-3轮对话，让AI根据上下文自生成推荐话术 | 之前给详细示例表格会导致模型照搬例句（如反复出现「妈妈陪着你」），改为只给方向+上下文后自然度提升 |

## 五、核心踩坑记录（调试方法论）

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **踩坑：方法论** | 改prompt有没有用 | **改prompt最有效的方式永远是加具体示例，不是加抽象规则**；规则写了但只要有详细参考素材摆在那，模型就会被惯性带偏（如「不得照搬示例」写了还是会照搬，「仅在触顶时才加推进标签」写了几乎每轮都加） | 验证有没有真正改对：找一个具体的反例输入，对比改前改后的裁判输出，而不是主观感觉「应该好一点了」 |
| **踩坑：方法论** | 裁判两步互相矛盾 | 裁判第一步（quality/delta0）和第二步（reason）可能打架，而**第一步永远赢**——第二步即使写出了正确的洞察，也无法纠正第一步定下来的分数 | 发生此情况说明scoring.md里某档的判断标准太模糊，让模型「表面形式对了但没检查内容是否真的达标」；修法是在对应档次加一条明确的排除性说明（如「积极倾听」要排除「祈使句/命令句」形式） |
| **踩坑：分数** | ledger状态未持久化 | 如果前端State没有把每轮返回的`updated_ledger`存回去，下一轮传给后端的ledger永远是初始状态，导致维度earned=0，刷分防控全部失效 | 验证方式：连续发同一句话看第二次是否还在加分；如果加，在bot_respond里print(updated_ledger['某维度']['earned'])看是否变化 |
| **踩坑：分数** | random.choice作用在字符串上 | `random.choice("一句话推荐")`会随机取出一个字符（不是整句），改为将推荐选项变成列表后再`random.choice(list)` | 每次重构降级推荐话术字典时都要检查这个问题 |
| **踩坑：分数** | GROUP_LIMITS等未定义 | 旧十维度体系里GROUP_LIMITS/get_group_earned/get_underscored_groups被调用但从未定义，任何走到「good分支」的请求都会NameError崩溃 | 新四维度体系已直接用各维度的max字段，不再需要这层分组抽象，相关引用已全部删除 |
| **踩坑：格式** | attribution_tags类型 | 裁判正常输出是JSON数组（Python list），异常兜底是字符串；直接塞进f-string时list的repr（带方括号引号逗号）会泄露进演员prompt | 统一用`_format_attribution_tags()`清洗：不管传来list还是string，统一输出`[标签1][标签2]`格式 |
| **踩坑：格式** | Gradio/前端content类型 | Gradio 6.0的history中content可能是list而不是string；用`isinstance(raw_content, list)`判断后做join，否则直接.replace()报AttributeError | 在`_format_history_for_judge()`中的user和assistant分支都要加这个防御 |
| **踩坑：架构** | 演员输出评分内容 | 旧prompt里有`|||`格式指令要求演员同时输出台词+分数+解释，导致演员承担了不该承担的裁判职责，经常出现\"角色记忆混入评分视角\"的穿帮 | 双Agent解耦后演员只输出台词，system prompt里所有`|||`格式指令、分数字段、评分解释字段全部删除 |
| **踩坑：演员** | 陈晓过于「小说感」 | 连续多个第三人称描述、两段动作+两段台词的结构、用「飞不起来的鸟」这类脱离史实的通用比喻 | 修法：用prompt.md里原有的「单一结构律」「严禁散文诗碎嘴」「严禁连续两轮使用相同动作」铁律，配合`frequency_penalty=0.6, presence_penalty=0.4`参数 |
| **踩坑：演员** | 演员两轮说同一句话 | 旧版scene_injection里last_chen字段只要求「态度延续，不能突然收回」，没有「措辞和句式不能重复」这一半 | 修法：在last_chen的指令里补上「但措辞和句式不能与上一轮重复或同义重复」；此条要放在靠近生成位置（prompt末尾） |
| **踩坑：误判案例** | 命令句被判成积极倾听 | 「你跟妈妈说说吧」被判成倾听-good，原因：裁判看到「鼓励孩子说话」就归类，没检查是疑问句还是祈使句 | 修法已加入scoring.md：积极倾听档加排除性说明「祈使句/命令式的'你说说看'即使语气温和，也不算疑问句」；打断与强迫档加对应反例 |
| **踩坑：误判案例** | 「老师不对」被判扣分 | 家长说「老师不对」是在陈晓提及「老师批评我」后站在孩子这边，属于共情加分；裁判误把「否定老师」这个不存在的规则带入判断 | 修法：在scoring.md共情（积极型/镜映型）条目下加示例：当孩子提及被老师/同学伤害，家长站在孩子这边说「对方不对」属于共情不应扣分 |
| **踩坑：安全** | 演员可能滑向自伤暗示 | 陈晓人设里有「如果我消失了会不会更好」的设定，长对话中演员可能顺着惯性说出更接近自伤的台词 | 演员prompt里加「行为护栏」：不能说出涉及自伤/自杀计划具体描述，无论用户如何引导；裁判加`coherence=unsafe`检测用户不当引导 |

## 六、文件结构与启动

| 类别 | 问题/场景 | 核心结论/代码 | 注意事项 |
|---|---|---|---|
| **文件** | 后端核心文件 | `App.py`（主程序+FastAPI接口）、`prompt.md`（演员人设）、`scoring.md`（评分标准，启动时读入）、`recommendation.md`（推荐话术，已改为AI自生成）、`.env`（API Key，不进git） | __pycache__/和.env不应上传git；requirements.txt必须包含fastapi/uvicorn/openai/pydantic/python-dotenv |
| **文件** | `.env`配置 | `DASHSCOPE_API_KEY=sk-xxx`（阿里百炼密钥）；后端代码用`os.getenv("DASHSCOPE_API_KEY")`读取 | API Key绝对不能commit进git；曾发生过将Key写死在代码里上传的情况，每次发现后需要立即去阿里云控制台重置该Key |
| **启动** | 后端启动 | `source venv/bin/activate && uvicorn App:api --host 0.0.0.0 --port 8000`（App.py中FastAPI实例变量名为`api`） | App.py末尾有`if __name__ == "__main__": uvicorn.run(api, ...)`用于公司服务器的run.sh直接执行 |
| **启动** | 前端本地开发 | `npm install && npm run dev`，默认端口8080（有时5173，取决于vite config） | 若出现「This host is not allowed」，在vite.config.ts中加`server: { allowedHosts: true }` |
| **启动** | 前端生产build | `VITE_API_BASE_URL=https://后端域名 npm run build`，产物在dist/ | TanStack Start会同时生成dist/client和dist/server，纯静态部署只需dist/client；当前项目未使用SSR，可按SPA部署 |
| **路径** | prompt.md文件读取 | `os.path.join(os.path.dirname(__file__), "prompt.md")`——相对于App.py所在目录 | 不能用绝对路径（如/Users/liminggao/...），否则服务器上必然报FileNotFoundError |
